# 需求文档：天眼查板块日期过滤模块

---

## 1. 模块概述

编写一个 Python 函数，接收 Playwright 的 `Page` 对象和两个日期，自动在页面中查找指定的业务板块，提取板块内表格中符合日期范围的行数据，并以结构化格式返回。

---

## 2. 函数签名

```python
def extract_sections_by_date(
    page: Page,
    date_start: str,
    date_end: str
) -> list[dict]
```

### 参数说明

| 参数 | 类型 | 说明 |
|---|---|---|
| `page` | `playwright.sync_api.Page` | 已加载目标页面的 Playwright Page 对象 |
| `date_start` | `str` | 日期范围起始，格式 `YYYY-MM-DD`，例如 `"2023-01-01"` |
| `date_end` | `str` | 日期范围结束，格式 `YYYY-MM-DD`，例如 `"2026-01-01"`，必须晚于 `date_start` |

### 入参校验要求

- 函数在执行前必须校验 `date_end > date_start`，否则抛出 `ValueError`，并附带说明信息。
- 日期格式不合法时同样抛出 `ValueError`。

## 3. 业务标识符列表

以下是页面中需要识别的业务板块，通过 HTML 属性 `data-dim` 唯一标识。

> ⚠️ **占位说明**：以下列表由开发者自行补充，根据实际页面结构填入所有需要处理的 `data-dim` 值。

```python
    "mergePunish",    # 【待填写】
    # ... 继续补充
]
```

## 4. 核心处理逻辑

### 4.1 遍历业务标识符

对 `TARGET_DIMS` 中的每一个值，执行以下流程：

```
在页面中查找 [data-dim="xxx"] 元素
        ↓
    找到 → 进入板块处理流程
    找不到 → 输出警告日志，跳过，继续下一个
```

找不到时的日志格式：
```
[WARN] 未找到业务标识符对应的板块元素: data-dim="xxx"
```

---

### 4.2 提取板块标题

在已定位的板块容器内，查找标题文字。定位规则如下：

- 遍历板块容器内所有子元素，取**第一个 `inner_text()` 不为空**的元素，其文本即为板块标题。不限定标签类型（`div`、`span`、`h3` 等均可命中），只要是容器内最前面出现的有效文本即视为标题。
- 若容器内找不到任何有效文本，则标题记为 `"未知板块"`，并输出日志：
  ```
  [WARN] 板块 data-dim="xxx" 未找到标题元素
  ```

---

### 4.3 提取表格数据

#### 步骤一：定位日期列

在板块容器的 `<table>` 元素中，采用以下策略定位日期列的列索引（从 0 开始计数）：

1. **优先**：读取 `<thead>` 中所有 `<th>` 的文本，找到包含"日期"**或**"时间"关键字的列（模糊匹配，如"公告日期"、"登记日期"、"注销时间"均可命中），记录其列索引。两个关键字哪个先匹配到就用哪个。
2. **降级兜底**：若 `<thead>` 不存在或没有任何 `<th>` 包含"日期"，则扫描 `<tbody>` 第一行所有 `<td>`，用正则 `\d{4}-\d{2}-\d{2}` 匹配，找到第一个命中的列，记录其列索引。
3. 若两种方式均未找到日期列，输出警告日志并跳过该板块：
   ```
   [WARN] 板块 "xxx" 未找到日期列，跳过
   ```

#### 步骤二：遍历所有数据行

对 `<tbody>` 中的**每一行**（`<tr>`）执行以下操作：

1. 提取该行所有 `<td>` 的文本，按列顺序存入列表，空白内容保留为空字符串。
2. 取出日期列对应的文本，尝试解析为日期对象（格式 `YYYY-MM-DD`）。
3. 若解析失败（内容为 `-`、空或格式不符），跳过该行，不计入结果，不输出日志。
4. 判断该日期是否满足：`date_start <= 该日期 <= date_end`（两端闭区间）。
5. 满足则将该行加入当前板块的结果列表。

> ⚠️ **注意**：遍历必须扫描所有行，不得因某行日期早于 `date_start` 而提前终止循环。

#### 步骤三：确定列字段名

列字段名从 `<thead>` 的 `<th>` 文本中获取，按顺序对应每一列。若某列表头为空，则字段名记为 `"列N"`（N 为从 1 开始的列序号，例如 `"列3"`）。

---

## 5. 日志规范

使用 `loguru` 输出日志，以下三种情况必须有对应日志输出：

| 情况 | 日志级别 | 示例 |
|---|---|---|
| 未找到 `data-dim` 对应元素 | `WARN` | `[WARN] 未找到业务标识符对应的板块元素: data-dim="mergePunish"` |
| 找到了日期列，但无符合日期范围的行 | `INFO` | `[INFO] 板块 "违规处理" 找到日期列，但无符合 2023-01-01 ~ 2026-01-01 范围的行` |
| 找到符合要求的行 | `INFO` | `[INFO] 板块 "违规处理" 共找到 3 行符合要求的数据` |

## 6. 返回值结构

函数返回一个列表，每个元素对应一个有数据的板块，结构如下：

```python
[
    {
        "section": "违规处理",         # str，板块标题
        "dim": "corpIllegals",         # str，data-dim 的值
        "rows": [                      # list，符合日期范围的行列表
            {
                "序号": "1",
                "公告日期": "2022-06-30",
                "处罚对象": "刘彦雷",
                "处罚类型": "出具警示函",
                "处罚金额": "-",
                "处理人": "中国证券监督管理委员会北京监管局",
                "操作": "详情"
                # 字段名从表头 <th> 文本获取，顺序与列一致
                # 若表头缺失则字段名为 "列1"、"列2" 等
            }
        ]
    }
]
```

### 特殊情况说明

- 若某个板块找到了日期列但没有符合范围的行，该板块**不出现**在返回列表中（只输出日志，不返回空的 `rows`）。
- 若某个板块未找到 `data-dim` 元素，同样**不出现**在返回列表中。
- 若所有板块均无符合数据，返回空列表 `[]`。

## 9. 注意事项

1. 所有 Playwright 操作需包裹在 `try/except` 中，单个板块出错不影响其他板块的处理。
2. 日期比较统一使用 `datetime.date` 对象，不做字符串比较。
3. 列文本提取使用 `.inner_text().strip()`，去除首尾空白。
4. 不使用 `> *` 直接子元素选择符，改用 `table`、`thead`、`tbody`、`tr`、`th`、`td` 等标签选择器，以兼容中间层级的变化。

## 10. 【给人类看】风险评估

> 本节不影响代码生成，供维护者了解脚本的整体逻辑和脆弱点，方便改版后快速定位问题。

### 模块整体业务逻辑概述

```
页面加载完成
      ↓
自动发现页面上所有带 data-dim 属性的板块容器（全量，不依赖预设列表）
      ↓
对每个板块：
    ├─ 检测是否为VIP付费板块
    │       ↓ 是 → 标记 vip: True，rows 为空，加入结果，跳过后续步骤
    │       ↓ 否 → 继续
    ├─ 提取板块标题（取容器内第一个有效文本）
    ├─ 定位日期列（表头关键字优先，正则兜底）
    ├─ 遍历所有数据行，筛选日期在范围内的行
    └─ 收集结果，输出日志
      ↓
返回所有板块的结果列表
```

---

### 各定位方式风险评估

| 定位目标 | 定位方式 | 风险等级 | 说明 |
|---|---|---|---|
| 业务板块容器（全量发现） | `page.locator("[data-dim]").all()` | ⭐ 低风险 | 不依赖预设列表，页面上有什么抓什么，新增板块自动覆盖 |
| 板块标题 | 容器内第一个 `inner_text()` 不为空的子元素 | ⭐ 低风险 | 不依赖标签类型，只依赖"标题在最前面"这一结构规律，极少失效 |
| VIP判断 | 容器全文关键字匹配（≥2个命中） | ⭐⭐ 低中风险 | 阈值宽松，漏判和误判概率均低；但依赖中文关键字，文案大改会失效 |
| 日期列（主） | `<thead>` 中含"日期"或"时间"的 `<th>` | ⭐⭐ 低中风险 | 双关键字兜底，改措辞概率已大幅降低 |
| 日期列（兜底） | 正则匹配 `<td>` 内容 `\d{4}-\d{2}-\d{2}` | ⭐⭐⭐ 中风险 | 若某列有类似日期格式的编号会误判；属最后手段 |
| 表格行数据 | `tbody tr td` 原生标签 | ⭐ 低风险 | 原生HTML表格结构，极少改动 |
| 列字段名 | `<th>` 文本内容按列顺序映射 | ⭐⭐ 低中风险 | 表头文字改动会导致返回字段名变化，影响下游依赖字段名的代码 |

---

### 最可能出问题的场景

1. **网站改版，`data-dim` 属性被移除或改名** → 自动发现机制找不到任何板块，函数返回空列表 `[]`，且不会有任何报错，需靠日志中"共发现0个板块"来察觉。这是最严重的失效场景，需重新分析页面结构。

2. **VIP付费提示文案全部换成英文**（如 "Unlock with membership"）→ 所有中文关键字均未命中，漏判为正常板块，进入表格提取流程后大概率返回空数据。应对方式：在 `VIP_KEYWORDS` 中补充英文关键字，如 `"Unlock"`、`"membership"` 等。

3. **正常板块的数据中恰好包含 ≥2 个VIP关键字**（极低概率）→ 误判为VIP板块，该板块被标记跳过，日志中有 `WARN` 可供排查。

4. **板块容器内标题前存在其他非空隐藏文本节点** → 会误取非标题内容作为板块名，数据本身不受影响，仅影响 `section` 字段的可读性。

5. **表头"日期"和"时间"措辞同时改变** → 日期列定位失败，触发正则兜底；若兜底也失败则跳过该板块，日志中有 `WARN` 提示。

6. **表头文字改动** → 返回结果中的字段名随之改变，下游若用字段名取值（如 `row["公告日期"]`）会报 `KeyError`，需同步更新下游代码。

## 11. 补充说明：多日期列的处理机制

### 背景

部分表格中可能存在多个包含"日期"或"时间"关键字的列，例如同时有"登记日期"和"注销日期"。原有逻辑只取第一个命中的列，遇到这种情况会漏判。

### 变更内容

**日期列定位阶段**，不再只记录第一个命中的列索引，而是收集所有命中的列索引，存入列表，例如：

```python
date_col_indices = [2, 6]  # 同时命中"登记日期"（第3列）和"注销日期"（第7列）
```

**行筛选阶段**，对每一行取所有日期列的值，逐一尝试解析为日期对象，只要其中**任意一个**满足 `date_start <= 该日期 <= date_end`，该行即视为合法行纳入结果。全部日期列均不在范围内（或均解析失败）时，才丢弃该行。

用一句话概括：**多个日期列取并集，任意一个命中即合法。**

### 示例

表格有两列日期：

| 登记日期 | 注销日期 |
|---|---|
| 2024-03-01 | 2021-05-01 |

用户输入范围：`2023-01-01 ~ 2026-01-01`

- "登记日期" 2024-03-01 ✅ 在范围内
- "注销日期" 2021-05-01 ❌ 不在范围内

结论：**该行合法，纳入结果。** 只要有一个日期命中，说明这条记录在目标时间段内有过关联，不应丢弃。

### 日志

当检测到某板块存在多个日期列时，输出一条提示日志：

```
[INFO] 板块 "xxx" 检测到多个日期列：["登记日期", "注销日期"]，将对所有日期列取并集筛选
```

## 12. 补充说明：分页表格抓取机制

### 背景

部分业务板块表格存在分页，即单页最多显示有限条数据（例如 10~20 行），用户期望抓取**前 N 条记录**（例如最多 20 条）。原有逻辑只处理当前页数据，分页数据未纳入。

### 思路

1. **检测分页按钮**

   * 在板块容器内查找“下一页”按钮，可使用如下定位方式（按现有风险偏好选择）：

     * `data-*` 或 `aria-label` 等属性定位（最优，低风险）
     * 文本匹配 `"下一页"`（中风险，易受文案改动影响）
     * 结合容器层级、子元素索引进行辅助定位
   * 若未找到下一页按钮，则认为该板块仅单页数据，直接处理当前页。

2. **数据变化判断**

   * 每翻页一次后，通过检测表格内容是否更新（例如首行日期或序号变化）确认页面已刷新，而非固定延时等待。
   * 避免因网络慢或动态加载导致的数据重复抓取。

3. **分页抓取逻辑**

   * 初始化 `collected_rows = []`
   * 遍历页面：

     ```text
     提取当前页所有符合日期条件的行 → 加入 collected_rows
     判断 collected_rows 是否已达用户指定上限（如 20 条）
         是 → 停止分页抓取
         否 → 检查是否存在下一页按钮
                 有 → 点击翻页，等待表格更新 → 继续循环
                 无 → 停止循环
     ```
   * 最终返回 `collected_rows[:N]`，确保不超过用户指定数量。

4. **日志规范**

   * 翻页操作每次输出：

     ```
     [INFO] 板块 "xxx" 翻页 2 / 共未知，已抓取 15 条数据
     ```
   * 达到抓取上限或无下一页时输出：

     ```
     [INFO] 板块 "xxx" 分页抓取完成，共抓取 20 条数据
     ```

### 注意事项

* 分页抓取必须与日期过滤逻辑结合，确保每页抓取的行均经过日期校验。
* 若分页按钮因文案改动或结构变化找不到，应回退为只抓取当前页并输出警告：

  ```
  [WARN] 板块 "xxx" 未找到下一页按钮，仅抓取当前页数据
  ```
* 避免因翻页过快导致数据重复或漏抓，可加入短时轮询检测表格更新或使用 Playwright 的 `wait_for_function` 等方法。

用一句话概括：**分页抓取按页顺序累积符合日期的行，直到达到最大条数或无下一页为止，并在每次翻页后检测表格更新以保证数据正确。**
