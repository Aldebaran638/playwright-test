# 网站初始化

当用户希望让 Codex 创建“网站初始化模块”时，使用这个 skill。

这个 skill 当前只支持“自动化登录型”的网站初始化。

## 适用范围

- 只支持自动化登录
- 当前版本不支持人工手动登录初始化
- 当前不支持以下情况：
  - 图片验证码
  - 滑块验证码
  - 短信验证码
  - 邮件验证码
  - 动态令牌验证
  - 任何需要人工完成第二步确认的登录流程

如果用户确认目标网站存在以上任一情况，就停止生成流程，并明确说明当前 skill 不支持这个网站初始化场景。

## 必需输入

在开始生成任何代码前，先收集以下信息：

- 目标网站 URL
- 完整可运行的 Playwright 登录代码
  预期来源是用户提供的、类似 `playwright codegen <url>` 生成并可运行的代码
- 是否存在当前不支持的验证码或人工二次验证
- 凭证来源
  选填，默认使用 `keyring`

当前版本不要求用户必须提供：

- 站点名称
- 登录成功判定
- 登录失败判定

如果目标网站没有独立登录页，也可以接受“从首页进入登录流程”的情况。通常用户提供的 Playwright 代码里已经体现了这一点。

## 交互流程

严格按以下顺序进行：

1. 先确认用户要建立的是“网站初始化模块”。
2. 询问登录类型。
3. 如果用户选择的不是“自动化登录”，就说明当前 skill 只支持自动化登录，并停止。
4. 明确提醒用户：当前不支持验证码和任何人工二次验证。
5. 向用户收集：
   - 目标网站 URL
   - 完整可运行的 Playwright 登录代码
   - 如果用户不想使用默认 `keyring`，再询问凭证来源
   - 用户确认该登录流程不涉及当前不支持的验证方式
6. 检查信息是否完备。
7. 只有在信息完备后，才开始生成代码。

## 生成规则

始终遵循 `MODULE_GENERATION_GUIDE.md`。

固定使用以下规则：

- 模块集群目录：`modules/site_init/`
- 初始化主入口函数名：`initialize_site`
- 初始化接口固定为：

```python
page = initialize_site(context)
```

- `initialize_site(context)` 接收上一个“浏览器上下文初始化”流程已经创建好的 `context`
- `initialize_site(context)` 返回一个 `page`
- 返回的 `page` 必须处于后续业务流程可以直接继续操作的状态

## 需要生成的文件

生成以下文件：

1. `tasks/` 下的 keyring 凭证写入任务文件
2. `modules/site_init/` 下的网站初始化模块
3. `tasks/` 下的网站初始化流程文件
4. `tests/site_init/` 下的测试文件

目录结构必须遵循 `MODULE_GENERATION_GUIDE.md`。

## Keyring 任务要求

把凭证写入脚本作为 `tasks/` 下的单独任务文件创建。

该文件应包含：

- 必要导入
- 顶部硬编码的全局凭证变量
- 用户只需要修改顶部这一段凭证信息
- 在 `main()` 中把凭证写入 keyring

除非用户明确要求别的凭证来源，否则默认使用 `keyring`。

## Codegen 代码使用边界

把用户提供的 Playwright 登录代码视为已经验证过的登录流程来源。

可以做的事：

- 对代码进行重构
- 删除录制产生的噪音步骤
- 提取可复用辅助函数
- 改造成符合项目规范的模块结构

不能做的事：

- 删除用户已经验证过的关键登录动作
- 在没有依据的情况下，私自改成另一套猜测的登录流程

## 验证职责

生成完成后，Codex 必须：

1. 先自检生成文件
2. 运行生成出来的测试文件
3. 向用户说明生成了什么、测试了什么

然后由用户进行真实网站测试。

如果用户反馈初始化模块有问题，Codex 应继续修正这个 `site_init` 模块。

## 交付检查清单

结束前确认：

- 生成文件符合 `MODULE_GENERATION_GUIDE.md`
- 模块文件位于 `modules/site_init/`
- 测试文件位于 `tests/site_init/`
- 初始化主入口为 `initialize_site(context)`
- `tasks/` 下已创建 site-init 任务和 keyring 任务
- 已运行生成的测试文件；如果没有运行，必须明确说明
