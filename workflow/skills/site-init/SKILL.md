# 网站初始化
当用户希望让 Codex 创建“网站初始化模块”时，使用这个 skill。

这个 skill 当前支持两个分支：
- 自动化登录型初始化
- 人工手动登录型初始化

## 通用规则
始终遵循 `MODULE_GENERATION_GUIDE.md`。

固定使用以下规则：
- 模块集群目录：`modules/site_init/`
- 初始化主入口函数名：`initialize_site`
- 初始化接口固定为：

```python
page = initialize_site(context)
```

- `initialize_site(context)` 接收上一阶段“浏览器上下文初始化”流程已经创建好的 `context`
- `initialize_site(context)` 返回一个 `page`
- 返回的 `page` 必须处于后续业务流程可以直接继续操作的状态

## 浏览器上下文衔接规则
网站初始化模块默认依赖已经构建好的浏览器上下文模块。

如果项目里还没有浏览器上下文初始化模块，Codex 在生成 `site_init` 之前，应先建立一套浏览器上下文模块，至少满足以下要求：
- 有一个可以被其他模块直接复用的浏览器上下文构建函数
- 该函数调用后可以直接拿到可用的 `context`
- 浏览器上下文支持配置模式和手动模式
- 终端等待用户选择输入模式时，要有倒计时显示
- 用户输入超时后，自动回落到配置模式
- 默认配置文件放在项目根目录下的 `data/other/browser_context_config.json`

## Cookie 复用规则
无论是自动化登录型还是人工手动登录型，网站初始化模块都要多做一件事：

1. 在前往目标首页之前，优先复用已经存在的 cookie
2. cookie 文件统一保存在项目根目录下的 `data/other/` 中
3. 如果用户没有额外指定 cookie 文件名，默认使用 `data/other/site_init_cookies.json`
4. 如果 cookie 文件存在且内容有效，就在打开首页前通过 `context.add_cookies(...)` 导入
5. 登录成功以后，再调用 `context.cookies()` 获取最新 cookie，并覆盖写回同一个文件
6. 如果 cookie 文件不存在、为空、格式错误或导入失败，不要中断初始化流程，记录日志后继续正常初始化

## 交互总流程
严格按以下顺序进行：

1. 先确认用户要建立的是“网站初始化模块”
2. 询问登录类型
3. 根据用户选择，进入“自动化登录型”或“人工手动登录型”分支
4. 收集该分支所需信息
5. 检查信息是否完备
6. 只有在信息完备后，才开始生成代码

## 自动化登录型

### 适用范围
- 只适用于程序可以自动完成登录动作的网站
- 当前不支持以下情况：
  - 图片验证码
  - 滑块验证码
  - 短信验证码
  - 邮件验证码
  - 动态令牌验证
  - 任何需要人工完成第二步确认的登录流程

如果用户确认目标网站存在以上任一情况，就停止自动化分支，并明确说明当前 skill 不支持这个自动化登录初始化场景。

### 必需输入
在开始生成任何代码前，先收集以下信息：
- 目标网站 URL
- 完整可运行的 Playwright 登录代码
  预期来源是用户提供的、类似 `playwright codegen <url>` 生成并可运行的代码
- 是否存在当前不支持的验证码或人工二次验证
- 凭证来源
  选填，默认使用 `keyring`

当前版本不要求用户必须提供：
- 站点名称
- 登录成功判定
- 登录失败判定

如果目标网站没有独立登录页，也可以接受“从首页进入登录流程”的情况。通常用户提供的 Playwright 代码里已经体现了这一点。

### 生成文件
自动化登录型需要生成以下文件：

1. `tasks/` 下的 keyring 凭证写入任务文件
2. `modules/site_init/` 下的网站初始化模块
3. `tasks/` 下的网站初始化流程文件
4. `tests/site_init/` 下的测试文件

### Keyring 任务要求
把凭证写入脚本作为 `tasks/` 下的单独任务文件创建。

该文件应包含：
- 必要导入
- 顶部硬编码的全局凭证变量
- 用户只需要修改顶部这一段凭证信息
- 在 `main()` 中把凭证写入 keyring

除非用户明确要求别的凭证来源，否则默认使用 `keyring`。

### Codegen 代码使用边界
把用户提供的 Playwright 登录代码视为已经验证过的登录流程来源。

可以做的事：
- 对代码进行重构
- 删除录制产生的噪音步骤
- 提取可复用辅助函数
- 改造成符合项目规范的模块结构

不能做的事：
- 删除用户已经验证过的关键登录动作
- 在没有依据的情况下，私自改成另一套猜测的登录流程

## 人工手动登录型

### 适用范围
- 适用于程序不负责输入账号密码，而是由用户自己在浏览器中完成登录
- 程序只负责打开页面、等待用户登录，并轮询检查是否已经达到“登录成功”状态

### 必需输入
在开始生成任何代码前，先收集以下信息：
- 网站首页地址
- 登录成功的检测标准

### 选填输入
- 等待用户登录的最大时长
  默认 `10min`
- 检测轮询间隔
  默认 `3s`

当前版本不要求用户提供：
- 未登录状态检测标准
- 站点名称
- 额外登录入口说明

### 登录成功检测标准
用户必须提供“登录成功的检测方法”。

这个方法不强制限定为某一种固定格式，可以是：
- selector 特征
- text 特征
- url 特征
- html 特征
- 其他足够清晰、且大模型判断为可实现的检测方式

如果用户给出的检测方法太模糊，无法稳定实现，就继续追问，直到大模型判断该检测方法明确、可实现、可测试。

### 生成功能要求
人工手动登录型的初始化模块内部逻辑应为：
1. 基于传入的 `context` 创建或接管页面
2. 在打开首页前，尝试导入 `data/other/site_init_cookies.json` 或用户指定的 cookie 文件
3. 打开首页
4. 提示用户手动登录
5. 按轮询间隔持续检查“登录成功标准”
6. 一旦满足成功标准，先更新 cookie 文件，再返回 `page`
7. 到达最大等待时间后报错

### 生成文件
人工手动登录型需要生成以下文件：

1. `modules/site_init/` 下的网站初始化模块
2. `tasks/` 下的网站初始化流程文件
3. `tests/site_init/` 下的测试文件

人工手动登录型不生成 keyring 任务。

## 验证职责
生成完成后，Codex 必须：
1. 先自检生成文件
2. 运行生成出来的测试文件
3. 向用户说明生成了什么、测试了什么

然后由用户进行真实网站测试。

如果用户反馈初始化模块有问题，Codex 应继续修正这个 `site_init` 模块。

## 交付检查清单
结束前确认：
- 生成文件符合 `MODULE_GENERATION_GUIDE.md`
- 模块文件位于 `modules/site_init/`
- 测试文件位于 `tests/site_init/`
- 初始化主入口为 `initialize_site(context)`
- 自动化登录型时，`tasks/` 下已创建 site-init 任务和 keyring 任务
- 人工手动登录型时，不生成 keyring 任务
- 在网站初始化流程里，已经加入 cookie 导入和登录成功后的 cookie 回写
- 已运行生成的测试文件；如果没有运行，必须明确说明
