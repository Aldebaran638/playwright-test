# 浏览器自动化环境模块工作流

## 1. 目标

这个工作流用于指导“浏览器自动化环境模块”的生成与决策。

当前阶段默认复用 `mintel` 项目中的浏览器上下文实现思路，只对参数和降级规则做标准化。

这里的重点不是立即启动浏览器，而是：

- 根据用户输入推断目标环境类型
- 对目标类型做一次真实兼容测试
- 如果失败，则按规则自动降级
- 最终得到一个可执行的浏览器环境方案


## 2. 用户输入参数

当前阶段只要求用户最多提供两个参数：

1. 浏览器可执行文件路径
2. 浏览器数据文件夹路径

说明：

- “浏览器数据文件夹路径”更准确地说，是自动化专用的浏览器用户数据目录
- 这份目录可以是用户真实环境的 copy 版
- 当前阶段不强制要求用户提供 `profile`
- 后续如果需要，可以默认使用 `Default`


## 3. 四种目标类型

### 情况 1

用户同时提供了：

- 浏览器可执行文件路径
- 浏览器数据文件夹路径

目标类型：

- `full_persistent`

解释：

- 使用用户指定浏览器
- 使用用户指定数据目录
- 持久化启动


### 情况 2

用户只提供了：

- 浏览器可执行文件路径

目标类型：

- `custom_browser_ephemeral`

解释：

- 使用用户指定浏览器
- 不使用用户数据目录
- 临时上下文启动


### 情况 3

用户只提供了：

- 浏览器数据文件夹路径

目标类型：

- `default_browser_persistent`

解释：

- 使用 Playwright 默认 Chromium
- 尝试复用用户提供的数据目录
- 持久化启动

注意：

- 这种情况下必须做真实兼容测试
- 因为这份数据目录未必和默认 Chromium 兼容


### 情况 4

用户两个都没提供

目标类型：

- `default_browser_ephemeral`

解释：

- 使用 Playwright 默认 Chromium
- 不复用任何本地数据目录
- 临时上下文启动


## 4. 核心原则

### 原则 1：先确定目标类型，再做兼容测试

程序不能只靠参数存在与否就直接宣布类型成功。

正确流程是：

1. 根据用户输入推断目标类型
2. 对这个目标类型做一次真实兼容测试
3. 测试成功，才真正采用该类型
4. 测试失败，则降级


### 原则 2：失败时按原因精细降级

不是所有失败都直接回退到最低档。

必须根据失败原因判断下一档应该降到哪里。


### 原则 3：每一步都要给出明确终端提示

程序必须告诉用户：

- 当前正在尝试哪种模式
- 为什么失败
- 已经降级到哪一种模式
- 最终成功使用了哪一种模式


## 5. 降级规则

### 从 `full_persistent` 降级

如果用户同时提供了浏览器路径和数据目录，但兼容测试失败：

#### 情况 A：浏览器程序不可用，但数据目录还在

例如：

- 浏览器路径不存在
- 浏览器根本无法启动

则降级到：

- `default_browser_persistent`

解释：

- 保留数据目录
- 改用 Playwright 默认 Chromium 继续尝试


#### 情况 B：数据目录不可用，但浏览器程序可用

例如：

- 数据目录不存在
- 数据目录损坏
- 数据目录不支持持久化启动

则降级到：

- `custom_browser_ephemeral`

解释：

- 保留用户浏览器
- 放弃该数据目录


#### 情况 C：两边都不行，或者无法区分失败原因

则降级到：

- `default_browser_ephemeral`


### 从 `custom_browser_ephemeral` 降级

如果用户只提供了浏览器路径，但启动失败：

则降级到：

- `default_browser_ephemeral`


### 从 `default_browser_persistent` 降级

如果用户只提供了数据目录，但默认 Chromium 与该目录不兼容，或者数据目录本身不可用：

则降级到：

- `default_browser_ephemeral`


### `default_browser_ephemeral`

这是最终兜底模式。

如果这一档仍然失败，则直接返回最终失败结果。


## 6. 为什么“只给数据目录”必须做真实兼容测试

因为：

- 用户的数据目录未必来自标准 Chromium
- 即使来自 Chromium 系，也未必能被 Playwright 默认 Chromium 正常复用
- 例如搜狗、第三方 Chromium 变体、不同版本内核，都可能产生兼容问题

因此，与其静态猜“像不像”，不如直接做一次真实启动测试。


## 7. 推荐返回结构

浏览器环境模块不应只返回浏览器对象。

推荐同时返回结构化决策信息，例如：

```python
{
  "requested_mode": "full_persistent",
  "resolved_mode": "custom_browser_ephemeral",
  "success": True,
  "used_fallback": True,
  "fallback_chain": [
    "full_persistent",
    "custom_browser_ephemeral"
  ],
  "reason": "user_data_dir is unavailable",
  "messages": [
    "...终端提示..."
  ]
}
```


## 8. 当前阶段是否可以直接落地

可以。

这套工作流已经足够成为浏览器自动化环境模块的第一版正式规则。

后续增强方向主要是：

- 更细的失败原因分类
- 更友好的终端提示
- 增加可选 `profile`
- 增加更多浏览器类型支持
